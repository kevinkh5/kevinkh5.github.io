GCP에서 Redis를 처음 띄울 때, 아무 고민 없이 Memorystore를 골랐다. 그 이유는 솔직히 Redis는 Memorystore를 사용하는 것이 보편적일 것이라고 생각했고(관리나 모니터링 등 기능이 지원하니까), VM에 직접 올릴 생각 자체를 못했다.

하지만 Memorystore의 가장 큰 단점으로는 외부 연결이 거의 불가능 하다는 것이었다

Memorystore는 매니지드 서비스라 내부 VPC 안에서는 편하지만, 외부에서 바로 붙을 수 없다.
문제는 내가 외부 서버랑도 연결해야 했다는 것… (장고서버가 AWS-EC2에 띄워져 있었음)

그래서 나온 해결책은

외부 연결 전용 VM을 하나 더 만들어서 통로처럼 쓰는 것이었다

가능은 하지만,
	•	VM 추가 비용 ↗
	•	구성 복잡도 ↗
	•	관리 포인트 ↗

사실 Memorystore 자체도 싸지는 않다. 특히 개발/테스트 환경에서는 VM에서 Redis 띄우는 게 훨씬 가성비 좋아보였다

최종 결론: Memorystore → VM으로 갈아탐

VM에 Redis를 올리고,
VPC 설정에서 내부 IP만 잘 맞춰주고,
Django 서버와 연결해보니…

✔ 연결 부드럽고
✔ 보안적으로도 깔끔하고
✔ 비용까지 줄어들 수 있었다




I initially set up Redis on GCP using Memorystore, but later switched to running Redis on a Virtual Machine.
To be honest, the main reason I started with Memorystore was simply because I didn’t consider using a VM at all—and it seemed like the “officially recommended” way to deploy Redis on GCP.

However, I soon ran into a major limitation: Memorystore doesn’t allow direct external access, which made integration much more difficult. On top of that, the cost turned out to be slightly higher compared to running Redis on a VM.
The workaround was to spin up an additional VM as a kind of proxy or gateway for external access, but that meant more resources, more money, and more operational complexity.

So I eventually migrated from Memorystore to a VM-based Redis setup.
Once I configured the VPC network and IP settings properly, my Django server connected smoothly. It felt secure enough, simpler to operate, and noticeably cheaper.
